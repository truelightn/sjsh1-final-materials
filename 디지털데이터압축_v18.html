손실 압축 (Lossy)
압축 시 일부 데이터를 영구 제거하여 용량을 크게 줄임. 디코딩해도 원본과 동일하게 복원 불가
대표 형식: JPEG, MP3, MPEG4
비손실 압축 (Lossless)
압축 과정에서 손실이 없어 디코딩 시 원본과 완전히 동일한 데이터로 복원 가능
대표 형식: ZIP, PNG, GIF, FLAC
압축률 = 압축 전 데이터 용량 ÷ 압축 후 데이터 용량
압축률이 높을수록 많이 줄었다는 의미  |  압축률 = 1이면 압축 효과 없음  |  압축률 < 1이면 오히려 커진 것
02
데이터 압축이 왜 필요할까?
저장 · 전송 · 환경 측면의 필요성
💾

저장 공간 절약

같은 크기의 저장 공간에 더 많은 데이터를 저장할 수 있어 비용과 효율이 좋아집니다

⚡

전송 속도 향상

같은 시간에 더 많은 데이터를 전송할 수 있어 스트리밍과 다운로드 속도가 빨라집니다

🌱

에너지·환경 보호

하드웨어 사용량과 에너지 소비를 줄여 전자 폐기물 감소에도 기여합니다

🎬 실생활 사례: 4K 영상 1분 분량은 압축 없이 약 1.5GB. 넷플릭스는 압축 기술로 이를 약 300MB로 줄여 스트리밍합니다.
🎵 음악 파일 1곡(WAV 무압축) ≈ 50MB → MP3 압축 후 ≈ 5MB (약 10배 감소)
📁 문서 파일 100개를 ZIP으로 압축하면 평균 60~70% 용량 절감 가능

이진수 RLE — 0과 1의 압축
이진 데이터에서의 반복 길이 압축과 추가 최적화
🔢 이진수(0과 1)에도 적용되는 RLE
① 기본 이진수 압축 예시

(14비트)
압축
1×5 0×5 1×1 0×1 1×2
(5쌍 → 10비트)
✅ 14비트 → 10비트로 압축 (1.4배)
이진수는 0과 1만 존재하므로, 어떤 숫자인지 굳이 표시하지 않아도 됩니다.

압축의 원조 — 모스 부호
빈도에 따라 길이가 다른 부호 — 허프만 압축의 조상
📡 압축의 원조 — 모스 부호 (Morse Code)
모스 부호는 1838년 새뮤얼 모스가 개발한 부호 체계로, 점(·)과 선(—) 두 가지 신호만으로 알파벳을 표현합니다.

07
허프만 압축 — 빈도수가 같을 때
균등 빈도에서의 허프만 트리와 고정 길이 코드의 관계
⚖️ 빈도수가 같을 때 허프만 트리는 어떻게 될까?
빈도수가 모두 같으면, 허프만 트리는 완전 균형 이진 트리에 가까운 형태가 됩니다.
모든 문자가 동일한 비트 수를 할당받으므로, 사실상 고정 길이 코드와 유사해집니다.
예시: ABCDABCDABCD

📌 균등 빈도 → 할당되는 코드 길이 모두 동일
A→00(2비트)
B→01(2비트)
C→10(2비트)
D→11(2비트)
균등 빈도의 허프만 트리 구조
결과: 원본 12자 × 8비트 = 96비트
압축 후: 12자 × 2비트 = 24비트 → 4배
💡 빈도가 동일하면 모든 잎이 같은 깊이에 위치 → 코드 길이 균등. 이 경우 허프만 코드는 고정 길이 코드(2비트)와 동일한 결과가 됩니다.
📊 빈도수가 일부만 같을 때는 어떻게 될까?
빈도수가 일부 문자끼리만 같고 나머지는 다를 때, 트리는 비대칭 형태가 됩니다.
빈도가 같은 문자끼리는 동일한 코드 길이를 받지만, 빈도가 다른 문자들 사이에는 코드 길이 차이가 생깁니다.

비대칭 허프만 트리 구조
✅ 빈도 차이 있을 때 — A(1비트)·B(3비트)·C/D(3비트)·E/F(4비트)
→ 빈도가 높을수록 짧은 코드, 트리가 오른쪽으로 치우침
⚖️ 빈도 같은 쌍 — C·D(2회→3비트), E·F(1회→4비트)
→ 같은 빈도끼리는 트리의 같은 레벨에 배치돼 코드 길이도 동일

JPEG 압축의 원리
인간 시각의 한계를 이용한 영리한 손실 압축 — RGB 표현부터 크로마 서브샘플링까지
단계 0 / 5
① 픽셀·RGB → ② YCbCr 채널 분리 → ③ 서브샘플링 → ④ 압축 효과 → ⑤ 정리
▶ 버튼을 눌러 JPEG 압축의 5단계를 순서대로 확인해보세요!
STEP 1 이미지는 숫자의 집합 — 픽셀과 RGB
비슷해 보이지만 숫자는 다른 두 픽셀
R: 216G: 103B: 81
픽셀 A
≈
R: 214G: 99B: 79
픽셀 B
두 픽셀은 값이 다르지만 사람 눈에는 거의 같아 보입니다.
JPEG는 이런 "비슷한 색"을 하나로 묶어 버립니다.
RGB 색상 모델
• 빨강(R) · 초록(G) · 파랑(B) 각각 8비트
• 픽셀 하나 = 3바이트 (24비트)
• 1920×1080 이미지 ≈ 6MB (비압축)
// 픽셀 한 개의 데이터
R: 0~255   G: 0~255   B: 0~255
// 총 경우의 수: 256³ = 16,777,216가지
STEP 2 RGB → YCbCr 변환 & 채널 분리 ⚠️ 색상 채널 손실 발생
변환 (예시 픽셀)

G 90
B 62
⟹

// 변환 공식
Y = 0.299R+0.587G+0.114B
Cb = 128-0.169R-0.331G+0.5B
Cr = 128+0.5R-0.419G-0.081B
4×4 픽셀 채널 분리 시각화
원본 컬러
4×4 = 16픽셀
Y — 밝기
4×4 유지 ✅

→ 압축 시 4픽셀↓

→ 압축 시 4픽셀↓
💡 Y 채널: 윤곽·질감·디테일이 모두 여기에.
그레이스케일이지만 이미지 품질의 핵심.
⚠️ Cb·Cr 채널: 눈이 색상 차이에 둔감.
2×2 블록을 하나로 묶어도 티가 안 남.

STEP 5 핵심 정리 — "눈이 못 느끼는 정보는 버린다"
픽셀 = RGB 숫자
모든 이미지는 픽셀의 격자. 픽셀 하나 = R·G·B 각 8비트. 비압축 시 용량이 매우 큼.
색공간 변환
RGB → YCbCr. 밝기(Y)와 색상(Cb, Cr) 분리. 인간 눈은 밝기에 훨씬 민감하다는 생리적 특성 활용.
Y 채널 유지
밝기 채널은 전체 해상도 그대로. 이미지의 윤곽·질감·디테일이 여기에 있기 때문.
서브샘플링 4:2:0
Cb·Cr을 2×2 묶기. 색상 데이터 75% 절감. 품질 저하 거의 없음.
여기까지 효과
크로마 서브샘플링만으로 전체 데이터 약 50% 감소.
